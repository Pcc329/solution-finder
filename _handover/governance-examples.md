# 治理判斷案例（本專案具體實例）

> 通用判斷框架見 `core/governance-checklist.md`，這份是套用在本專案的實際案例，
> 供未來遇到類似情境時參考「上次是怎麼判斷的」。

## 資料分級：三層架構

- 有公司資料（基本）< 有方案（第二層）< 有案例（第三層，最完整）
- 資料庫擴張（ETL 進更多筆數）本身不是護城河；案例庫（顧問實戰經驗的結構化沉澱）才是

## 案例：外部參展名單是否匯入 Companies 表（2026-07-22）

- 情境：同事提供一份網路爬蟲取得的參展廠商名單，共 929 家不重複公司，其中 57 家與現有資料庫重疊
- 判斷：57 家重疊公司留存參考；872 家全新公司**不匯入**
- 理由：爬蟲來的粗資料，非本平台驗證過的資料，貿然併入會膨脹筆數、稀釋方案/案例的覆蓋率指標，且需要長期維護對應治理成本
- 處理方式：全程唯讀比對，零 Schema 異動，沒有新增欄位、沒有寫入任何記錄

## 案例：MCP Server 整合的觸發條件

三個明確條件，缺一不推進：
1. 案例庫需要 AI 同時查詢多個來源
2. Supabase 遷移完成之後
3. 坤達平台團隊主動提出跨系統整合需求

主管順口問及 MCP 進度，不等於觸發條件 3 成立——區分「順口關心」跟「正式提出需求」。

## 案例：對同事的回饋用詞（2026-07-22）

- 情境：資料庫裡有 48 家公司連結到方案資料，來源多為雲市集、政府計畫公開登錄資料，其中 25 筆明確來自「新創嚴選網」
- 判斷：不能講「我們跟這些公司合作過」，只能講「我們對這些公司在雲市集/政府計畫上登記的方案有掌握」；新創嚴選網來源的可以講「入選新創嚴選網優質方案商」（背書性質，不是合作關係）
- 理由：「新創嚴選」是中小企業處主辦的公開平台，不是本院自營計畫，「合作過」容易讓對方誤以為有直接業務往來，一旦被追問對接窗口反而傷害可信度

## 案例：Supabase Security Advisor 的 CRITICAL 項目（2026-07-22）

- 情境：階段二 RLS 上線後，Security Advisor 標出 `Security Definer View`（companies_public、contacts_masked 兩個 view）跟 `RLS Enabled No Policy`（contacts 表）
- 判斷：立即修正，不等下一個階段
- 理由：View 預設用建立者權限執行，可能繞過底層表的 RLS，這類問題不會在功能測試中被發現，只能靠專門的資安掃描抓出來；RLS 開啟但沒有政策會直接擋掉所有人（含管理員），必須補上明確政策
